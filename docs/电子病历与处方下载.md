**【模块定位与技术选型】**
本模块负责医疗隐私数据的安全展示与凭证固化。
* **权限控制：** 强制实施基于会话（Session/Token）与数据所有者（Owner）的双重鉴权。
* **PDF 生成引擎：** 推荐使用 **FreeMarker (模板引擎) + Flying Saucer (HTML 转 PDF) + iText (底层驱动)**。绝对禁止用原生的 Java 字符串拼接去硬写 PDF 坐标！采用“编写 HTML 模板 -> 注入数据渲染 -> 转换为 PDF”的现代工程化流派。

**一、 数据库表结构规范 (MySQL)**
为了支撑该业务，你需要至少三张具有强外键逻辑的物理表：

1.  `medical_record` (电子病历主表)
    * 包含：`id`, `order_no` (关联预约单), `user_id` (患者ID), `doctor_id`, `symptoms` (主诉症状), `diagnosis` (确诊结果), `created_at`。
    * **索引要求：** 必须建立 `idx_user_id`，所有面向患者的查询必须带上 `user_id`。
2.  `prescription` (处方主表)
    * 包含：`id`, `record_id` (关联病历), `user_id`, `status` (0:有效, 1:已作废), `created_at`。
3.  `prescription_item` (处方明细表)
    * 包含：`id`, `prescription_id`, `medicine_name` (药品名称), `dosage` (单次剂量), `usage_instruction` (用法，如“口服，一日三次”)。

**二、 电子病历查看 (EMR View) - 零信任架构要求**

患者在前端 JSP 页面点击“查看病历”时，通过 AJAX 发起请求。

* **入参过滤：** 接收 `recordId`。
* **强制鉴权拦截 (Filter/Interceptor)：**
    必须从当前 HTTP Session (或 JWT) 中提取出真实的 `current_user_id`。**绝对不能信任前端传来的 userId。**
* **JDBC 安全查询 (SQL 级别隔离)：**
    在 DAO 层执行查询时，SQL 语句必须写死 `user_id` 过滤条件：
    `SELECT * FROM medical_record WHERE id = ? AND user_id = ?`
    如果查询结果为空，说明该病历不存在，或者**该用户正在试图越权查看别人的病历**，此时系统必须记录安全审计日志，并向前端返回 HTTP 403 / 404 错误。

**三、 处方 PDF 生成与下载 (PDF Generation)**

这是一个极其容易写出烂代码的地方。请严格按照以下架构分层实现：

**1. 模板准备层 (Resources)**
* 在 Spring Boot 的 `src/main/resources/templates` 目录下，创建一个 `.ftl` (FreeMarker) 模板文件（如 `prescription_template.ftl`）。
* 在这个模板里用标准的 HTML 和内联 CSS 写好处方的样式（包括诊所 Logo、网格线、患者信息占位符 `${patientName}`、处方明细的循环渲染 `<#list items as item>` 等）。
* **专家警告：** PDF 引擎默认不支持中文！你必须在服务器或项目中放置中文字体文件（如 `simsun.ttf`，宋体），并在程序中显式加载，否则生成的中文全是一堆 `####` 乱码。

**2. 数据聚合层 (Service)**
* 开启 JDBC 只读事务 (`@Transactional(readOnly = true)`)。
* 联合查询出 `medical_record`、`prescription` 和 `prescription_item`。同样，**必须携带 `user_id = ?` 进行越权防护**。
* 将数据封装成一个 Java `Map` 或 `DTO` 对象。

**3. 渲染与输出层 (Controller & HttpServletResponse)**
不要先在服务器磁盘上生成物理 PDF 文件再去下载（这会产生大量磁盘垃圾，且引发文件 IO 冲突）。必须采用**内存流式直传（Streaming）**。

```java
// Controller 层核心逻辑伪代码示范
@GetMapping("/api/prescription/download")
public void downloadPrescriptionPdf(@RequestParam Long prescriptionId, HttpServletRequest request, HttpServletResponse response) throws Exception {
    
    // 1. 获取当前安全上下文中的 userId
    Long currentUserId = SecurityUtils.getUserIdFromSession(request);
    
    // 2. 聚合数据 (Service 层必须校验 userId)
    Map<String, Object> dataModel = prescriptionService.getPrescriptionData(prescriptionId, currentUserId);
    if (dataModel == null) {
        response.sendError(HttpServletResponse.SC_FORBIDDEN, "无权访问");
        return;
    }

    // 3. 设置 HTTP 响应头，告诉浏览器这是一个待下载的 PDF
    response.setContentType("application/pdf");
    response.setHeader("Content-Disposition", "attachment; filename=\"prescription_" + prescriptionId + ".pdf\"");

    // 4. FreeMarker 渲染 HTML 字符串
    String htmlContent = freeMarkerService.renderTemplate("prescription_template.ftl", dataModel);

    // 5. Flying Saucer / iText 将 HTML 转为 PDF 并直接写入 Response 的 OutputStream
    try (OutputStream out = response.getOutputStream()) {
        ITextRenderer renderer = new ITextRenderer();
        // 关键：加载中文字体，否则中文变乱码
        renderer.getFontResolver().addFont("fonts/simsun.ttf", BaseFont.IDENTITY_H, BaseFont.NOT_EMBEDDED);
        renderer.setDocumentFromString(htmlContent);
        renderer.layout();
        renderer.createPDF(out);
        out.flush();
    } catch (Exception e) {
        // 记录系统错误日志
        throw new RuntimeException("PDF生成失败", e);
    }
}
```

### 架构考点总结

在实际开发或项目评审中，重点强调以下两点以体现架构思维：
1.  **“我在所有获取隐私数据的 SQL 级别都加入了 Session UserId 的强制绑定，从根本上杜绝了水平越权漏洞。”**
2.  **“处方的 PDF 生成我没有采用低效的硬盘落地方案，也没有用字符串拼接，而是采用了 HTML 模板渲染加 `OutputStream` 内存直推客户端的技术，既保证了样式的易维护性，又提升了高并发下的 I/O 性能。”**
