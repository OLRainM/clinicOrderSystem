### 诊所用户管理与权限控制系统核心需求

**【模块定位与架构红线】**
本模块负责系统底层的基础账号体系与鉴权分发。
**架构红线**：严禁将高频查询的登录鉴权数据与低频庞大的业务档案数据揉在同一张表中。抛弃传统的有状态 `HttpSession`，强制采用 **Cookie (HttpOnly) + Redis Token** 的无状态分布式鉴权模型。密码严禁明文或普通 MD5 存储，必须使用 BCrypt。

**一、 数据库表结构设计 (MySQL)**
核心思想：账号与档案分离 (Account-Profile Separation)。

```sql
-- 1. 系统账号主表 (极其精简，只用于登录和鉴权)
CREATE TABLE `sys_user` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `phone` VARCHAR(11) UNIQUE NOT NULL COMMENT '手机号作为唯一登录凭证',
    `password_hash` VARCHAR(64) NOT NULL COMMENT 'BCrypt哈希值',
    `role_type` TINYINT NOT NULL COMMENT '1:患者, 2:医生, 3:管理员',
    `status` TINYINT NOT NULL DEFAULT 1 COMMENT '1:正常, 0:禁用',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX `idx_phone` (`phone`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 2. 患者业务档案表 (关联 sys_user)
CREATE TABLE `patient_profile` (
    `user_id` BIGINT PRIMARY KEY COMMENT '既是主键，也是 sys_user.id 的外键',
    `real_name` VARCHAR(32) NOT NULL,
    `id_card_no` VARCHAR(18) UNIQUE NOT NULL COMMENT '身份证号',
    `gender` TINYINT COMMENT '1:男, 2:女',
    `birthday` DATE,
    `emergency_contact` VARCHAR(11)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 3. 医生业务档案表 (关联 sys_user)
CREATE TABLE `doctor_profile` (
    `user_id` BIGINT PRIMARY KEY,
    `real_name` VARCHAR(32) NOT NULL,
    `department_id` BIGINT NOT NULL COMMENT '所属科室',
    `title` VARCHAR(32) NOT NULL COMMENT '职称：如 主任医师',
    `introduction` TEXT COMMENT '医生简介',
    INDEX `idx_dept` (`department_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**二、 安全防线与认证链路设计**
采用 Cookie (HttpOnly) + Redis Token 鉴权模型，兼顾 JSP 渲染与 AJAX 请求。

**1. 注册与密码加盐：**
必须使用 Spring Security 提供的 `BCryptPasswordEncoder`。注册时插入 `sys_user` 与对应的档案表，必须放在同一个 `@Transactional` 事务中。

**2. 登录发牌 (Login & Token Generation)：**
* 校验手机号与 BCrypt 密文。
* 成功后生成强随机 UUID 作为 Token。
* **Redis 存储：** 将 `SET auth:token:{UUID} {user_id} EX 7200` (2小时过期) 存入 Redis。
* **下发凭证：** Servlet 将 Token 写入名为 `CLINIC_TOKEN` 的 Cookie。**必须设置 `HttpOnly = true` 与 `Path = /`**，彻底封杀 XSS 窃取可能。

**3. 统一权限拦截器 (HandlerInterceptor)：**
所有非公开路由（`/user/**`, `/doctor/**`, `/admin/**`, `/api/**`）必须经过全局拦截器。
* 拦截器从 Cookie 中提取 `CLINIC_TOKEN`。
* 拿着 Token 去 Redis 校验：若不存在或已过期，AJAX 请求返回 `401 Unauthorized`，JSP 页面请求重定向至 `/login`。
* 若校验成功，对 Redis Token 续期（保活），并将 `user_id` 注入 `HttpServletRequest` 属性中（`request.setAttribute("current_user_id", userId)`），供后续 Controller 安全取用。

**三、 权限控制 (RBAC) 落地**
基于角色实现路由级与接口级控制。

* **自定义注解：** 定义 `@RequireRole(int value)`，`value` 对应角色类型（1:患者, 2:医生, 3:管理员）。
* **拦截器校验：** 在鉴权拦截器或单独的权限拦截器中，获取当前请求方法的注解。若当前用户的 `role_type` 与注解要求不符，直接阻断请求并返回 `403 Forbidden`。
* **隔离越权：** 在业务逻辑（如查看病历、取消预约）中，必须强制使用从拦截器解析出来的 `current_user_id` 作为 SQL 查询条件，严禁信任前端传入的 userId。

**四、 前端交互逻辑**
* **登录动作：** `/login` 页面提交表单，AJAX 获取成功状态后，执行 `window.location.href` 重定向至对应角色的工作台。
* **后续请求：** 浏览器会自动携带 Cookie，后续的所有 JSP 页面加载与 AJAX 异步请求均实现无缝鉴权。
