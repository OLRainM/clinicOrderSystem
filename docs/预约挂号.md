**【模块定位与红线警告】**
本模块处理患者的核心交易链路（预约、支付、取消、改签）。
**架构红线**：严禁在“改签”业务中执行物理删除或简单的时段 `UPDATE`。订单状态机只能单向流转，改签必须遵循“创建新单、继承状态、作废旧单”的不可变数据（Immutable Data）原则。

**一、 核心状态机定义**
`appointment_order` 表中的 `status` 字段必须包含以下完备的状态枚举：
* `0`: 待支付（占号锁定中，10分钟倒计时）
* `1`: 已支付（预约成功）
* `2`: 已取消（用户主动取消，或超时未支付由系统取消）
* `3`: 已改签（该订单已被废弃，被新的改签订单替代）

*(可选扩展字段：`original_order_no` 用于记录改签产生的新订单是从哪个旧订单演变而来的，方便财务追溯)*

**二、 核心业务流转要求**

**1. 预约挂号 (Reservation) - 强化防御**
* **前置校验：** Controller 层必须拦截同一用户的高频异常请求（可使用 Spring AOP 结合 Redis Token 桶限流）。
* **占号逻辑：** 严格复用上文定义的 Lua 脚本 (`reserve_quota.lua`) 进行 Redis 库存原子扣减与防重复锁号。
* **落库逻辑：** Redis 扣减成功后，通过自定义线程池 `@Async` 或 RabbitMQ 发起异步 JDBC 事务，插入状态为 `0` (待支付) 的订单。

**2. 取消预约 (Cancellation) - 安全回滚**
用户主动取消分为两种场景，必须在 Service 层做严格的 `if-else` 分支处理：
* **场景A：取消“待支付(0)”订单**
    * 校验订单拥有者。
    * 开启 JDBC 事务：修改订单状态 `0 -> 2`，执行 MySQL 库存 `available_quota + 1`。
    * 事务提交后，清理 Redis：`INCR` 归还该时段号源，`HDEL` 解除用户的防刷锁定。
* **场景B：取消“已支付(1)”订单**
    * 必须先调用第三方支付（如微信/支付宝）或内部财务系统的**退款 API**。
    * 退款申请成功/受理后，开启 JDBC 事务：修改订单状态 `1 -> 2`，MySQL 库存 `+ 1`。
    * 事务提交后，执行 Redis 库存归还逻辑（同上）。*注意：如果距离出诊时间小于设定阈值（如2小时），应拒绝取消。*

**3. 改签逻辑 (Rescheduling) - 事务的艺术（核心难点）**
改签接口 (`/api/order/reschedule`) 接收 `old_order_no` 和 `new_slot_id`。必须严格按照以下步骤执行：

* **步骤一：预检旧单**
    通过 JDBC 校验旧订单状态必须为 `1` (已支付)，且属于当前用户。
* **步骤二：试探性锁定新时段（Redis 层）**
    调用 Lua 脚本尝试扣减 `new_slot_id` 的 Redis 号源。
    * *分支 1 (失败)*：如果新时段号源不足，直接向前端返回“新时段号源已满，改签失败，您的原预约依然有效”。**流程安全终止。**
    * *分支 2 (成功)*：新号源扣减成功，进入复杂的数据库替换事务。
* **步骤三：强一致性替换（JDBC 事务内执行）**
    开启 `@Transactional` 事务：
    1.  **创建新单：** 插入一条新订单，关联 `new_slot_id`，状态直接设为 `1` (已支付) —— *继承了原订单的资金状态*。
    2.  **作废旧单：** 将 `old_order_no` 的状态从 `1` 更新为 `3` (已改签)。
    3.  **置换库存：** 新时段 MySQL 库存 `- 1`；旧时段 MySQL 库存 `+ 1`。
* **步骤四：释放旧时段缓存**
    事务提交成功后，对旧时段的 Redis 进行操作：`INCR` 归还旧时段号源，清除旧时段的用户锁。
