### 后台数据统计模块核心需求 (Java 技术栈版)

**【模块定位与架构红线】**
本模块负责为诊所管理层提供宏观的数据决策支持（如科室就诊量、医生接诊率）。
**架构红线**：严禁在前端请求到达时，直接在核心业务表（`appointment_order`）上执行实时的全表聚合查询（`COUNT` / `GROUP BY`）。必须采用**定时任务离线聚合**策略，将计算压力转移到凌晨低谷期。

**一、 聚合统计表设计 (MySQL)**
建立专门的统计结果表，用于承载前端页面的极速查询。

1.  `stat_daily_department` (科室每日就诊量聚合表)
    * 字段：`id`, `stat_date` (统计日期), `department_id`, `department_name`, `visit_count` (就诊数量), `updated_at`。
    * 索引：建立联合唯一索引 `uk_date_dept (stat_date, department_id)`。
2.  `stat_daily_doctor` (医生每日接诊率聚合表)
    * 字段：`id`, `stat_date`, `doctor_id`, `doctor_name`, `total_appointments` (总预约数), `completed_appointments` (实际完结/接诊数), `reception_rate` (接诊率，精确到小数点后两位), `updated_at`。
    * 索引：建立联合唯一索引 `uk_date_doc (stat_date, doctor_id)`。

**二、 离线聚合任务 (Spring Task Scheduled)**
利用 Spring Boot 自带的定时任务模块，在业务低谷期自动执行复杂聚合计算。

* **技术实现：** 使用 `@Scheduled(cron = "0 0 2 * * ?")`（每天凌晨 2 点执行）。
* **处理逻辑：**
    1.  统计“昨天”的数据：确定时间区间（如 `2026-06-13 00:00:00` 到 `23:59:59`）。
    2.  利用 `JdbcTemplate` 从 `appointment_order` 和 `medical_record` 等业务表中，通过原生的 `GROUP BY` 聚合出昨天的科室总量和医生接诊数据。
    3.  利用 MySQL 的 `INSERT ... ON DUPLICATE KEY UPDATE` 语法，将计算结果批量（Batch Update）写入上述的两张 `stat_daily_*` 表中，保证任务的可重入性（即使跑失败了重新跑也不会产生重复数据）。

**三、 报表接口输出 (Controller & Service)**
后台前端面板（如基于 ECharts 渲染的 JSP 页面）请求统计数据时，接口的逻辑必须保持极简。

* **API 设计：** `/admin/api/stats/department` 与 `/admin/api/stats/doctor`。
* **查询逻辑：** 接收前端传入的时间范围（如近 7 天），Service 层直接对 `stat_daily_department` 等聚合表执行简单的 `SELECT SUM(visit_count) ... WHERE stat_date BETWEEN ? AND ? GROUP BY department_id`。
* **性能保障：** 由于查询的是已经按天聚合好的微型表（每天每个科室/医生只有 1 条记录），即便查询一整年的数据，底层扫描的行数也仅在千级别，接口响应时间可严格控制在 50ms 以内，彻底保护了主交易链路的安全。
